# WebGuard 系统架构文档

**版本**：V1.0  
**适用范围**：WebGuard 仓库内前端、后端、浏览器插件、部署配置与 Codex 协同开发  
**文档定位**：本文件定义 WebGuard 的目标架构、模块边界、数据流与关键非功能约束。它不是宣传文档，而是实现与重构时必须遵守的技术基线。

---

# 1. 背景与目标

WebGuard 是一个“**Web 平台为主、浏览器插件为辅**”的恶意网站检测与安全预警系统。

系统目标不是单纯“扫一下网址”，而是构建一个完整的闭环：

1. 用户通过 Web 平台完成登录、插件安装、插件绑定、历史记录查看、策略配置和报告查看。
2. 浏览器插件在访问网页时执行轻量预处理与云端联动检测。
3. 后端完成统一风险评估、报告生成、策略管理和审计记录。
4. 数据最终沉淀到数据库，并可被统计、查询、复盘与运营。

---

# 2. 总体架构原则

## 2.1 一级原则

- **Web 平台是主产品面**，负责完整的用户体验与管理能力。
- **插件是辅助运行面**，负责浏览期的实时扫描、预警、拦截与跳转。
- **后端是唯一可信判定源**，所有高价值业务规则与风险结论以后端为准。
- **数据库是最终真相**，缓存与本地状态只是性能优化层。

## 2.2 二级原则

- 统一接口规范，避免前端、插件、后端“三套语言”。
- 所有关键行为都必须可审计、可追踪、可回溯。
- 优先保证边界清晰，其次再做局部性能优化。
- 当前仓库允许存在过渡态实现，但目标架构不能被过渡态绑架。

# 3. 当前仓库现状与目标架构的关系

## 3.1 当前仓库三端结构

```text
frontend/   -> Web 前端
backend/    -> FastAPI 后端
extension/  -> 浏览器插件
```

## 3.2 当前已观察到的现状

### 前端
- React 18 + TypeScript + Vite
- React Router
- Axios
- ECharts
- Tailwind CSS
- 当前页面已经包含：产品首页、登录、插件安装引导、用户工作区、管理员页面、报告页等

### 后端
- FastAPI
- SQLAlchemy
- Alembic
- Pydantic v2
- 已有多个 API 模块：auth、scan、reports、records、stats、plugin、rules 等
- 检测主流程已具备：
  `FeatureExtractor -> RuleEngine -> DeepSeekAnalysisService -> Detector`

### 插件
- Manifest V3
- TypeScript
- 背景脚本、内容脚本、popup、options、warning 页面齐备
- 已具备扫描、警告页跳转、存储配置等基础链路

## 3.3 当前仓库存在的主要漂移

当前代码和配置存在几类不一致：

1. **数据库口径已收敛到 PostgreSQL**：
   - `docker-compose.yml`、根 `.env.example` 和 `backend/.env.example` 均以 PostgreSQL 作为当前目标数据库；
   - Schema 演进使用 Alembic；
   - SQLite 只允许作为 CI/单元测试轻量配置出现，不作为正式运行数据库。

2. **鉴权已切换到真实 Web 账号体系**：
   - Web 平台只走真实账号密码登录、真实注册、refresh/logout；
   - 旧的开发登录入口已移除，插件绑定机制仍按独立阶段推进。

3. **部署与开发说明没有完全统一**：
   - README、环境变量、容器配置之间语义存在偏差。

## 3.4 架构结论

本项目的**目标架构**为：

- Web：React + TypeScript + Vite
- Backend：FastAPI + PostgreSQL + Redis（作为目标能力）
- Extension：Manifest V3 + TypeScript
- API：统一 REST + JSON
- 鉴权：生产级会话/令牌体系

同时允许短期兼容当前仓库的一部分过渡实现，但**所有新的工作都应朝目标架构收敛**。

# 4. 技术栈定版

## 4.1 前端（Web 平台）

### 定版栈
- React 18
- TypeScript
- Vite
- React Router v6
- Axios
- Tailwind CSS
- ECharts

### 建议增强但不强制一次性引入
- TanStack Query：管理服务端状态
- React Hook Form：表单状态控制
- Zod：前端参数校验
- Zustand：用于少量 UI 全局状态

### 设计理由

前端主要承担以下职责：

- 产品首页与营销说明
- 用户登录与会话管理
- 插件安装与绑定流程
- 用户扫描记录与报告查看
- 个人白名单/黑名单/信任域名策略
- 管理后台（规则、域名、模型、统计、插件、用户）

这个形态明显更适合标准 SPA + 组件化页面组织，而不是把大量逻辑压进插件页面中。

---

## 4.2 后端（服务端）

### 定版栈
- Python 3.11+（仓库当前 README 提到 3.14，可接受，但团队应固定一个开发版本）
- FastAPI
- Pydantic v2
- SQLAlchemy 2.x
- Alembic
- PostgreSQL（目标主数据库）
- Redis（目标缓存/限流/异步协调能力）
- Uvicorn / Gunicorn + Uvicorn Worker

### 设计理由

后端要承担：

- 鉴权与授权
- URL 检测编排
- 风险策略与规则管理
- 用户级策略（白名单/黑名单/临时信任）
- 报告生成
- 插件事件同步
- 审计与统计

FastAPI + Pydantic + SQLAlchemy 组合适合强契约 API、快速演进与清晰分层。

---

## 4.3 插件（Extension）

### 定版栈
- Chrome / Chromium Extension
- Manifest V3
- TypeScript
- Popup / Options / Warning 页面可使用轻量 TS 或 React 化组织
- Service Worker 作为后台编排中心
- `chrome.storage.local` 作为本地持久层

### 设计理由

插件承担“轻而快”的实时动作：

- 感知当前页面 URL
- 拉取本地缓存配置
- 调后端检测接口
- 根据结果提示/拦截/跳转
- 打开 Web 报告页

插件不能承担复杂管理后台能力，否则产品重心会跑偏。

# 6. 系统交互与数据流

# 6.1 主链路：插件实时检测

### Step 1：用户访问网页
用户在浏览器中打开目标网站。

### Step 2：插件感知页面变化
Service Worker 或导航监听逻辑捕获当前 URL。

### Step 3：插件本地预判
先检查：
- 是否浏览器内部页
- 是否本地白名单域名
- 是否命中临时信任
- 是否命中短 TTL 缓存

若满足安全放行条件，则跳过远程调用。

### Step 4：插件请求后端
插件调用检测接口，上传：
- URL
- 页面标题（可选）
- referrer（可选）
- 插件实例标识
- 插件版本
- 用户访问令牌或绑定后的鉴权凭证

### Step 5：后端执行检测流水线
后端执行：
- 输入标准化
- 特征抽取
- 规则判断
- DeepSeek 大模型语义研判
- 风险汇总
- 记录持久化
- 报告生成

### Step 6：插件执行动作
根据结果：
- SAFE -> 静默放行
- SUSPICIOUS -> 轻提醒或建议查看报告
- MALICIOUS -> warning 拦截页 / 阻断动作

### Step 7：用户进入 Web 报告页（可选）
用户可从插件跳转到 Web 平台查看详细报告。

---

# 6.2 Web 管理与查看链路

### Step 1：用户登录 Web 平台
完成认证，进入用户工作区或管理员工作区。

### Step 2：前端获取数据
前端调用记录、报告、规则、策略等接口。

### Step 3：后端鉴权和查询
后端根据用户身份与资源归属决定是否可访问。

### Step 4：前端可视化展示
展示：
- 历史记录
- 风险评分
- 命中规则
- 个人域名信任状态
- 统计结果

# 7. 鉴权与授权设计

# 7.1 当前状态

当前仓库仍保留开发态 mock login，主要用于本地快速看页面。该能力只能算开发辅助工具。

# 7.2 目标设计

## Web
推荐：
- Access Token：短时有效
- Refresh Token：HttpOnly Cookie
- 前端仅在内存保留 Access Token

## 插件
推荐：
- 插件实例绑定
- `plugin_instance_id`
- `plugin_bind_token` 或等价绑定凭证
- 插件以绑定关系换取短期访问权限

## 后端
后端要支持：
- 用户身份校验
- 用户角色校验
- 插件实例校验
- 插件实例与用户归属关系校验
- 审计日志记录

# 7.3 授权边界

- 普通用户只能看自己的记录、报告和策略
- 管理员才能管理全量规则、域名、用户、模型、全量记录
- 插件不能天然拥有管理权限，必须依附用户授权或绑定关系

# 8. 数据存储设计

# 8.1 核心实体

建议/现有模型可围绕以下领域收敛：

- user
- plugin_instance
- scan_record
- scan_report
- domain_rule / user_strategy
- blacklist / whitelist / domain_list
- rule_config
- model_version
- feedback
- audit_log

# 8.2 数据库选择

## 目标数据库
PostgreSQL

理由：
- 支持更清晰的 JSON 字段能力
- 更适合复杂查询、统计和可演进结构
- 已在 `docker-compose.yml` 中部分体现

## 迁移原则

所有新文档、新部署方案、新环境示例应围绕 PostgreSQL + Alembic 收敛。测试中可以使用 SQLite 降低 CI/单元测试成本，但正式运行、部署模板和答辩口径不应把 SQLite 描述为运行数据库。

# 9. 缓存与一致性

# 9.1 插件本地缓存

可缓存：
- 最近检测过的域名结果
- 白名单快照
- 临时信任项
- 插件配置

必须带：
- TTL
- 版本号
- 明确失效策略

# 9.2 Redis 目标用途

- 热门 URL / 域名检测结果缓存
- 插件 bootstrap 配置缓存
- 限流计数器
- 短期会话与任务状态

# 9.3 一致性原则

- DB 为最终真相
- Redis 为性能层
- 插件本地缓存为体验层
- 所有缓存都必须可失效、可重建

# 10. CORS 与跨端协同

## Web 调后端
只允许白名单域名源。

## 插件调后端
必须显式允许扩展源。

## 禁止项
- 生产环境 `CORS_ORIGINS=*`
- 因为调试方便而永久放大跨域范围

# 12. 当前开发约束总结

在当前仓库上开发或指导 Codex 时，必须始终记住：

1. Web 是主产品，不要把复杂流程继续推到插件里。
2. 后端是可信中心，插件只做轻量编排和展示。
3. 数据库口径保持 PostgreSQL + Alembic，SQLite 仅限 CI/单元测试。
4. 任何新 API 都必须遵守统一契约。
5. 任何新功能都必须考虑将来上线，而不是只考虑本地能跑。

这就是 WebGuard 的架构基线。

## Current AI Detection Architecture

WebGuard 当前采用规则引擎 + DeepSeek 大模型语义研判。浏览器插件采集页面访问与交互特征，后端规则引擎生成可解释行为风险信号，并在命中高风险条件时调用 DeepSeek 分析页面语义诱导、品牌冒充、支付、验证码、钱包和潜在攻击意图。最终风险分在 DeepSeek 成功返回 `used` 时按 `behavior_score * 0.45 + deepseek_score * 0.55` 融合；未配置 DeepSeek、未触发、超时或异常时自动退回规则引擎兜底。

当前主检测链路只保留规则引擎 + DeepSeek 大模型语义研判。

DeepSeek 不替代规则引擎、黑白名单或外部威胁情报。管理员后台 AI 配置页是 DeepSeek / 火山方舟接入主路径，`.env` 中的 `DEEPSEEK_API_KEY` 仅作为数据库未保存 API Key 时的 fallback。
