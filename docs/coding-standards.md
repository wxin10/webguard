# WebGuard 生产级代码规范

**版本**：V1.0  
**适用范围**：backend / frontend / extension / tests / docs  
**文档定位**：本文件定义 WebGuard 代码编写、重构、审查与自动代理生成代码时必须遵守的底线规则。

---

# 1. 最高原则

WebGuard 的代码必须是：

- 可维护的
- 可测试的
- 可审计的
- 可扩展的
- 可回滚的

禁止把“能跑”当成“可交付”。

---

# 2. 通用规则

## 2.1 禁止项

禁止以下行为：

- 教程式玩具实现直接进入主干
- 未经设计的大范围复制粘贴
- 吞异常不处理
- 直接在 UI 层写复杂业务编排
- 接口返回结构临时拼凑
- 不写迁移只改模型
- 以 mock 流程冒充正式能力

## 2.2 代码风格底线

- 命名清晰，不允许模糊缩写泛滥
- 函数职责单一
- 模块边界明确
- 参数与返回值类型明确
- 所有公共行为都要可读、可追踪

---

# 3. 后端规范

# 3.1 分层规范

## Router / API 层
负责：
- 接收请求
- 参数解析
- 调用 service
- 输出统一响应

禁止：
- 写业务编排
- 写数据库操作细节
- 写复杂条件分支

## Service 层
负责：
- 业务编排
- 检测流程
- 策略判断
- 报告组装

禁止：
- 依赖 FastAPI 的 Request/Response 细节做核心逻辑
- 在一个 service 混入多个不相关领域

## Model / Repository 层
负责：
- ORM 模型定义
- 查询与持久化

禁止：
- 把 HTTP 响应、权限判断、业务文案写进数据层

---

# 3.2 Schema 与校验规范

- 所有请求体必须使用 Pydantic schema
- 所有响应体必须使用稳定 DTO 或经过统一包装
- URL、域名、分页、排序、枚举、时间字段必须显式校验
- 禁止“前端会保证”的假设

---

# 3.3 异常处理规范

必须具备：
- 统一异常基类
- 业务异常类型
- 参数异常处理
- 认证异常处理
- 权限异常处理
- 未知异常兜底

### 输出要求
- HTTP 状态码正确
- JSON 结构统一
- 不泄露内部堆栈

### 禁止
- `except Exception: pass`
- 无上下文的返回 `{"detail": "error"}`
- 任何地方都直接 `raise HTTPException` 替代领域异常设计（少量边缘场景除外）

---

# 3.4 日志规范

## 必须包含的上下文
- request_id
- path
- method
- user_id（若有）
- plugin_instance_id（若有）
- business_code
- latency

## 日志级别
- INFO：正常关键流程
- WARNING：可恢复问题、风控触发、兼容降级
- ERROR：明确失败
- CRITICAL：系统性故障

## 禁止记录
- 明文密码
- 完整 token
- 敏感隐私全文

---

# 3.5 数据库规范

- 所有表结构变化必须通过 Alembic 管理
- 主键命名与风格统一
- 高频查询字段要建索引
- 唯一性约束必须落到数据库层
- 审计字段统一：`created_at`、`updated_at`
- 如启用软删除，要有清晰规范

---

# 3.6 配置规范

- 配置必须通过环境变量或明确配置文件读取
- 禁止把密钥写死在源码里
- 配置项命名统一、语义明确
- 配置变更要更新 `.env.example` 和文档

---

# 4. 前端规范

# 4.1 目录职责

建议保持：
- `pages/`：页面级组件
- `components/`：通用组件
- `services/`：API 调用封装
- `contexts/`：少量上下文
- `types/`：共享类型
- `utils/`：纯工具函数

# 4.2 页面规范

页面组件负责：
- 布局组织
- 调用 hooks / services
- 状态展示

禁止：
- 直接把大量 Axios 调用散落在 JSX 中
- 把复杂业务逻辑写成组件内巨大 `useEffect`

# 4.3 服务端状态规范

- 服务端数据不要混成纯本地状态
- 同类接口统一封装
- API 错误处理方式统一
- DTO 类型统一定义

# 4.4 表单规范

- 提交前做基础校验
- 错误提示要有明确文案
- 后端校验错误要能显示到 UI

# 4.5 UI 状态规范

每个关键页面必须考虑：
- loading
- empty
- error
- partial data

禁止白屏式失败。

---

# 5. 插件规范

# 5.1 架构规范

- Background Service Worker 负责编排
- Popup / Options / Warning 负责页面 UI
- Storage 封装集中管理
- API 调用封装集中管理

# 5.2 安全降级规范

当后端异常时，插件必须：
- 避免浏览器卡死
- 避免无限重试
- 给用户可理解的最小提示
- 在高风险拦截逻辑中保持保守处理策略

# 5.3 权限规范

- `manifest.json` 中权限最小化
- host_permissions 不要过度放大
- 后续若改权限，必须同步文档与风险说明

# 5.4 缓存规范

- 本地缓存必须带 TTL
- 配置项必须可清理
- 不要在多个文件里各自维护同一份策略状态

---

# 6. 测试规范

# 6.1 后端

至少覆盖：
- schema 校验
- 核心 service
- 核心接口
- 关键权限判断
- 检测主链路

# 6.2 前端

至少覆盖：
- 关键页面渲染
- 登录/路由保护
- 关键交互与异常展示

# 6.3 插件

至少覆盖：
- URL 处理逻辑
- 存储逻辑
- 风险结果路由行为
- warning 页面关键行为

---

# 7. 文档同步规范

以下变化必须同步文档：
- 路由变化
- 鉴权变化
- 启动命令变化
- 环境变量变化
- 插件与 Web 的跳转规则变化
- 数据库口径变化

代码改了、文档没改，不算完成。

---

# 8. Review 审查清单

每次提交前至少检查：
- 有没有跨层污染？
- 有没有直接硬编码？
- 有没有异常被吞掉？
- 有没有测试或至少可验证步骤？
- 有没有更新文档？
- 有没有把过渡态写成永久态？

这份规范是 WebGuard 所有后续代码工作的最低标准。
