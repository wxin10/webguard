# 09-系统维护说明

## 配置维护

常用后端环境变量包括：

| 变量 | 说明 |
|---|---|
| `DATABASE_URL` | PostgreSQL 数据库连接地址 |
| `CORS_ORIGINS` | 允许访问后端的前端地址 |
| `JWT_SECRET` | JWT 签名密钥 |
| `DEBUG` | 调试模式开关 |
| `ENABLE_DEV_AUTH` | 开发鉴权兼容开关 |
| `ENABLE_RUNTIME_SCHEMA_GUARD` | 运行时建表兼容开关 |
| `REFRESH_TOKEN_COOKIE_SECURE` | Refresh Token Cookie 是否只允许 HTTPS |
| `DEEPSEEK_API_KEY` | DeepSeek API Key 兜底配置 |
| `DEEPSEEK_BASE_URL` | DeepSeek 接口地址 |
| `DEEPSEEK_MODEL` | DeepSeek 模型名称 |
| `DEEPSEEK_ENABLED` | AI 分析启用方式 |

服务器部署时应关闭开发兼容开关，使用强随机密钥，并配置 HTTPS。

## 数据库维护

数据库结构通过 Alembic 维护。升级数据库：

```powershell
cd backend
alembic upgrade head
```

新增数据库字段或表结构后，应生成并检查迁移脚本，确保生产数据库可重复升级。

## 账号维护

首次部署可使用脚本创建管理员账号：

```powershell
cd backend
$env:WEBGUARD_SEED_USERNAME = "admin"
$env:WEBGUARD_SEED_PASSWORD = "<强密码>"
$env:WEBGUARD_SEED_ROLE = "admin"
python -m app.scripts.seed_dev_user
```

管理员也可以在 Web 平台中创建、禁用、启用用户或重置密码。

## 规则维护

管理员可在后台维护检测规则。维护规则时应注意：

- 规则名称和描述应清晰表达检测意图。
- 权重和阈值需要结合测试样本调整。
- 停用规则优先于直接删除，便于回溯。
- 新规则上线前应使用规则测试功能验证命中效果。

## 黑白名单维护

全局白名单用于明确可信域名，全局黑名单用于明确阻止域名。维护时应避免过宽泛的域名规则，防止误伤正常访问。

## AI 配置维护

管理员可在 AI 配置页维护 DeepSeek API Key、base_url、模型名称和启用状态。配置完成后应执行连通性测试。AI 服务不可用时，系统仍会使用规则引擎完成检测。

## 插件维护

插件维护重点包括：

- 保持插件 API 地址与后端部署地址一致。
- 根据部署域名调整插件访问权限。
- 定期检查插件实例列表，撤销不再使用的实例。
- 检查插件事件上报是否正常。

## 日常检查

建议定期检查：

- `/health` 后端健康状态。
- 数据库连接和 Alembic 迁移状态。
- 检测记录增长是否正常。
- 管理员反馈处理是否及时。
- AI 配置是否可用。
- 前端和插件构建是否通过。

## 常用命令

后端测试：

```powershell
cd backend
python -m pytest
```

前端检查：

```powershell
cd frontend
npm run lint
npm run build
```

插件构建：

```powershell
cd extension
npm run build
```
